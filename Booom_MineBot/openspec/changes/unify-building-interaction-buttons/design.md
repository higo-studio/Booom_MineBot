## Context

第一轮反馈后，维修站和机器人工厂已经被视为建筑/设施占位，`R` 也被建筑模式占用。但当前玩家可见交互仍不完整：维修和生产从属机器人保留在表现层方法与测试辅助入口中，正常操作里没有明确按钮可以触发。结果是玩家能看到机器人工厂，却不知道如何生产从属机器人；维修站也存在同类入口不统一问题。

本设计把“建筑提供可点击主动作”作为表现层统一模式：玩家靠近可交互建筑时，HUD 显示建筑交互按钮；点击按钮后调用对应规则服务。规则真相仍在 `BaseOpsService`、`RobotFactoryService`、建筑占位和经济服务中，UI 只负责发现附近建筑、展示按钮和转发意图。

## Goals / Non-Goals

**Goals:**

- 让维修站和机器人工厂都通过统一的建筑交互按钮执行动作。
- 让玩家靠近建筑时能从 HUD 上直接看到可执行动作和成本/结果反馈。
- 保留现有规则服务，不改变维修、生产机器人、资源扣除或机器人生成结算。
- 让按钮状态遵守当前模式锁定：升级、失败、标记模式和建筑放置模式下不误触发建筑动作。
- 为后续建筑扩展预留同一套“建筑主动作”展示和回调模式。
- 用 PlayMode 测试覆盖按钮路径，而不是继续只测隐藏方法调用。

**Non-Goals:**

- 不新增全局快捷键，不恢复旧的 `R` 维修或 `B` 造机器人按键。
- 不实现复杂建筑详情面板、建筑升级、拆除、队列生产或多动作菜单。
- 不把建筑交互规则挪到场景对象或 Collider 上作为玩法真相。
- 不重做建筑定义、建筑放置服务或从属机器人自动模式。

## Decisions

### 1. 建筑交互入口放在 HUD 中，而不是绑定新按键

玩家靠近维修站或机器人工厂时，HUD 显示一个建筑交互区域，包含当前可执行动作按钮。例如“维修站：维修”和“机器人工厂：生产机器人”。按钮点击调用 `TryRepairAtStation` 或 `TryBuildRobotAtFactory`。

选择理由：

- 用户明确要求“统一为与建筑的交互按钮”。
- 当前键位已经被自由移动、探测、标记模式和建筑模式占用，继续加快捷键会让输入语义变乱。
- UGUI 按钮路径能被 PlayMode 测试直接验证。

备选方案：新增 `F` 通用交互键。

放弃原因：它可以作为未来补充，但本轮目标是可见按钮；只加键位不能解决“玩家看不到如何交互”的问题。

### 2. 建筑交互按钮由表现层根据逻辑位置刷新

`MinebotGameplayPresentation` 继续通过 `IsNearRepairStation()`、`IsNearRobotFactory()` 判断可用建筑动作，并刷新按钮显隐。按钮不读取场景 Collider 或 GameObject 名称来决定规则结果。

选择理由：

- 保持“逻辑网格是规则真相”的项目约束。
- 维修站和机器人工厂当前已经有稳定的逻辑位置和规则服务。
- 实现范围小，不需要先设计完整建筑动作注册表。

备选方案：给每个建筑 prefab 加 MonoBehaviour 并从碰撞范围发现动作。

放弃原因：会把规则入口分散到场景对象上，并且和当前建筑占位服务的权威模型冲突。

### 3. 第一版使用固定设施动作映射，后续再扩展为数据驱动动作表

第一版只需要维修站和机器人工厂两个动作，因此可在表现层建立明确映射：

- `repair-station` / 维修站：维修，调用 `TryRepairAtStation`
- `robot-factory` / 机器人工厂：生产从属机器人，调用 `TryBuildRobotAtFactory`

后续若新增更多建筑动作，再抽成 `BuildingInteractionDefinition` 或在 `BuildingDefinition` 上挂动作元数据。

选择理由：

- 当前 MVP 只有两个据点动作，抽象过早会拖慢实现。
- 固定映射可以快速修复玩家不可达的机器人生产入口。
- 后续扩展点明确，不阻塞本轮。

### 4. 按钮可用性遵守模式锁定

建筑交互按钮在以下状态不可执行：

- 升级面板打开或存在待选升级
- GameOver
- 当前处于 Marker 模式
- 当前处于 Build 模式

选择理由：

- 防止 UI 点击同时触发地图模式或建筑动作。
- 与第一轮 `GameplayInteractionMode` 设计保持一致。

## Risks / Trade-offs

- [Risk] 按钮只显示最近设施，玩家同时靠近维修站和工厂时可能不知道哪个优先。→ Mitigation: 第一版可同时显示两个按钮，或按现有两个设施独立显隐；测试覆盖同时可见时互不影响。
- [Risk] 维修站和工厂后续变成多格建筑后，当前 `ManhattanDistance` 靠近判定可能不够精确。→ Mitigation: 本轮保留现有范围判定；后续可改为基于建筑 footprint 的相邻格查询。
- [Risk] UI 按钮和测试辅助方法可能出现行为分叉。→ Mitigation: 按钮只调用现有 `TryRepairAtStation` / `TryBuildRobotAtFactory`，不复制规则逻辑。
- [Risk] 建筑模式面板和建筑交互按钮在屏幕右侧互相拥挤。→ Mitigation: 建筑交互按钮作为独立小面板放在 HUD 下方或右侧低位，Build 模式时隐藏。

## Migration Plan

1. 在表现层新增建筑交互按钮容器和两个按钮。
2. 将按钮点击绑定到现有维修/生产规则入口。
3. 更新 HUD 提示文案，明确“靠近建筑后点击按钮”。
4. 将 PlayMode 测试从直接调用 `input.Repair()` / `input.BuildRobot()` 改为点击 UI 按钮路径。
5. 保留旧方法作为测试辅助和内部调用入口，避免破坏现有规则测试。

## Open Questions

- 第一版按钮是否显示资源成本文本，还是只通过点击失败反馈提示“金属不足”。
- 后续建筑如果有多个动作，是扩展成一个小动作列表，还是每个建筑只允许一个主动作。
