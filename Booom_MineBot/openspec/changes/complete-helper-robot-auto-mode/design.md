## Context

原始策划案 revision 123 中，`5.7 从属机器人` 明确要求：从属机器人用于处理低风险、重复性的挖掘工作，不承担风险判断职责，不会挖掘玩家已标记岩壁，无法使用探测，受到致命伤害后损毁并掉落金属；MVP 至少实现自动模式，目标选择为最近可挖目标、避开标记区、无事可做时待机。`5.3 炸药、探测与标记` 进一步强调机器人只能执行玩家已经判断过风险的挖掘任务，玩家负责判断，机器人负责执行。`5.8 地震波与危险区` 要求机器人若在地震结算时处于危险区则直接被消灭。

当前实现已经有以下基础：

- `RobotState` 记录位置和存活状态。
- `RobotFactoryService` 能消耗金属生产机器人。
- `RobotAutomationService.TrySelectNearestSafeMineTarget` 能筛选最近目标，但目前目标筛选会读取隐藏 `HasBomb`，这等于给机器人隐藏炸药透视能力。
- `RobotAutomationService.StepToward` 只做单步直线移动，不能绕开障碍，也没有持续 tick、挖掘执行、奖励结算、目标失效重选或待机状态。
- `MinebotGameplayPresentation` 能显示从属机器人，但不展示机器人状态，也没有驱动完整自动挖掘闭环。
- `WaveSurvivalService` 已支持地震结算时摧毁危险区中的机器人并掉落资源。

本设计目标是把从属机器人从“能被生产并显示”推进到“能自动执行策划定义的低风险挖掘闭环”，同时避免引入非 MVP 的指令模式或复杂 AI 系统。

## Goals / Non-Goals

**Goals:**

- 建立从属机器人自动模式的明确状态：待机、移动、挖掘、受阻、损毁。
- 让机器人按固定 tick 自动选择目标、移动到目标附近、挖掘相邻岩壁并结算奖励。
- 将“低风险”定义为基于玩家可见/已表达的信息，而不是读取隐藏炸药真相。
- 避开玩家标记格、危险区格、不可达目标和超出钻头能力的目标。
- 目标被玩家挖掉、被标记、进入危险区或路径失效时，机器人能重选目标或待机。
- 与现有地震波系统衔接：危险区内机器人地震结算时损毁并掉落资源。
- 在 HUD 或场景表现中反馈机器人当前状态，便于玩家理解自动化是否有效。
- 用 EditMode 测试覆盖确定性规则，用 PlayMode 烟雾测试覆盖主场景表现。

**Non-Goals:**

- 不做指令模式、任务队列、区域派工、手动路线规划或多机器人协同分工。
- 不做复杂行为树、第三方 AI/FSM 框架、NavMesh、DOTS/ECS 或物理寻路。
- 不做机器人背包、返航卸货、充电、维修或独立升级系统。
- 不改变玩家探测规则、炸药生成规则或地震波主循环。
- 不让机器人自动使用探测，也不让机器人自动替玩家标记危险格。

## Decisions

### 1. 机器人安全判断只使用公开状态，不读取隐藏炸药真相

自动模式的目标资格应基于以下公开或玩家表达状态：

- 目标格是可挖岩壁。
- 目标格未被玩家标记。
- 目标格当前不处于地震危险区。
- 目标硬度不高于当前允许的机器人钻头能力。
- 存在可达的相邻可通行落脚格。
- 目标在配置的搜索距离内。

目标选择不得使用隐藏 `GridCellState.HasBomb` 作为避险条件。隐藏炸药只应在机器人实际挖掘时被结算；如果玩家没有标记真实炸药，机器人误挖后会承担后果。

选择理由：策划案强调玩家负责判断、机器人负责执行。读取 `HasBomb` 会让机器人比玩家更了解隐藏信息，破坏扫雷式风险判断。

备选方案：继续用 `HasBomb` 保证机器人永远不踩雷。放弃原因是这会让机器人拥有超出策划边界的风险判断能力，也会削弱标记和探测的玩法价值。

### 2. 使用轻量状态机，而不是行为树

`RobotState` 增加自动模式状态和目标信息：

```text
RobotState
├─ position
├─ isActive
├─ activity: Idle / Moving / Mining / Blocked / Destroyed
├─ targetPosition?
├─ lastStatusReason
└─ actionTimer
```

`RobotAutomationService` 负责每个 tick 对单个机器人执行一步决策：

```text
TickRobot(robot, context)
  if robot inactive -> Destroyed
  if target invalid -> clear target
  if no target -> select nearest eligible target or Idle
  if adjacent to target -> Mine
  else -> move one step toward staging cell
```

选择理由：MVP 只需要一种自动模式，状态枚举足以支持测试和 HUD 展示。行为树或通用 FSM 框架会增加过多工具成本。

备选方案：无状态每帧重选目标。放弃原因是表现层难以解释机器人正在做什么，目标被阻断或待机的反馈也会变差。

### 3. 用 BFS 选择“目标相邻落脚格”，不做路径缓存

机器人不能直接走到岩壁格内，而是需要走到目标岩壁的相邻可通行格。目标选择流程：

1. 枚举搜索半径内的可挖目标。
2. 对每个目标找相邻可通行落脚格。
3. 用 BFS 从机器人当前位置计算最近落脚格路径距离。
4. 按路径距离、目标曼哈顿距离、坐标顺序做确定性 tie-break。

移动时只执行路径的下一步，不缓存长期路径；下一 tick 重新验证目标和路径。

选择理由：地图是离散网格，BFS 简单、可测试、足够应对 12x12 级别 MVP 地图。每 tick 重算能自然适配玩家挖掘、标记、地震危险区变化。

备选方案：继续直线 `StepToward`。放弃原因是遇到墙体、边界或已挖通道转弯时容易卡住。

### 4. 复用挖掘规则，但把“挖掘者”从玩家状态中解耦

当前 `MiningService.TryMine` 绑定 `PlayerMiningState`。本变更应抽出共享方法，例如：

```text
TryMineFrom(GridPosition actorPosition, HardnessTier drillTier, GridPosition target, out ResourceAmount reward)
```

玩家挖掘继续通过 `GameSessionService.Mine` 处理玩家伤害、奖励和事件；机器人挖掘通过自动模式服务调用共享挖掘逻辑，结算地形与奖励。

如果机器人挖到未标记隐藏炸药：

- 机器人损毁。
- 触发爆炸地形结算。
- 不给玩家结算“直接挖到炸药”的直接伤害；玩家仍可能在后续地震或危险区规则中受到影响。
- 掉落或回收资源使用现有机器人损毁资源配置。

选择理由：这样能保持挖掘门槛、地形变更和奖励逻辑一致，同时避免把玩家专属伤害逻辑错误套到机器人身上。

备选方案：为机器人复制一份挖掘逻辑。放弃原因是硬度、奖励和炸药结算会分叉，后续调参容易不一致。

### 5. 自动模式由运行时服务驱动，表现层只调用和刷新

推荐在 `GameSessionService` 或新增轻量协调入口中提供：

```text
TickRobots(float deltaTime)
```

该入口内部调用 `RobotAutomationService`，并在机器人移动、挖掘、损毁或待机状态变化时触发 `StateChanged` / `RewardGranted` 或机器人专用事件。`MinebotGameplayPresentation.Update` 可以像当前地震 tick 一样调用该运行时入口，但不直接修改机器人规则状态。

选择理由：保持“规则状态由服务拥有、场景层负责表现”的项目架构。后续如果增加非场景运行宿主，也可以复用同一自动模式入口。

备选方案：在 `MinebotGameplayPresentation` 里直接寻路和挖矿。放弃原因是表现层会拥有玩法真相，违反项目架构边界，也难以写 EditMode 测试。

### 6. 机器人参数进入配置，不散落硬编码

优先扩展现有 `GameBalanceConfig` 或新增 `RobotAutomationConfig`，至少包含：

- 目标搜索半径。
- 行动间隔。
- 机器人可挖硬度策略：默认跟随玩家当前钻头等级，或配置固定上限。
- 机器人损毁回收资源：可继续复用 `WaveConfig.RobotRecycleDrop`，如语义不清则迁到统一机器人配置。

选择理由：策划案仍有“机器人行为边界”和数值待定项，参数必须方便在 Inspector 和测试中调整。

备选方案：保留常量。放弃原因是后续调节自动化强度时会频繁改代码。

### 7. HUD 反馈保持最小但可读

表现层需要显示或汇总：

- 活跃机器人数量。
- 工作中/待机/受阻数量。
- 最近一次机器人收益或损毁反馈。
- 机器人 Sprite 随位置刷新；受阻或待机可以通过轻量色彩/文本反馈体现。

选择理由：玩家需要知道自动化是否在生效，否则机器人只是在地图上“摆着”。但 MVP 不需要复杂机器人面板。

备选方案：只移动 Sprite 不显示状态。放弃原因是待机/受阻时玩家无法理解原因，会误判为 bug。

## Risks / Trade-offs

- [Risk] 不读取 `HasBomb` 后，机器人可能挖到玩家未标记的炸药，看起来像“机器人不够聪明”。→ Mitigation：HUD 文案明确机器人只执行、不判断风险；测试覆盖“未标记炸药可导致机器人损毁”。
- [Risk] BFS 每 tick 重算在大地图上有性能成本。→ Mitigation：MVP 地图尺寸小，先用搜索半径限制；后续需要大地图时再做路径缓存。
- [Risk] 机器人奖励即时进入玩家资源，可能让自动化收益偏强。→ Mitigation：行动间隔、搜索半径和可挖硬度进入配置，后续可调。
- [Risk] 机器人触发炸药是否伤害玩家存在设计灰区。→ Mitigation：MVP 先不结算玩家直接伤害，只结算机器人损毁和爆炸地形变化；如后续策划要求爆炸范围伤害，再单独提案。
- [Risk] 升级选择期间如果机器人继续工作，会破坏“升级打断正常循环”的手感。→ Mitigation：当 `ExperienceService.HasPendingUpgrade` 或玩家死亡时暂停机器人自动 tick。

## Migration Plan

1. 扩展自动化数据模型和配置字段，保持旧 `RobotState` 构造方式可用。
2. 抽出玩家/机器人共享的挖掘方法，确保现有玩家挖掘测试不变。
3. 实现目标资格判断、BFS 落脚格选择和单机器人 tick。
4. 在运行时服务中加入多机器人 tick、奖励/损毁事件和暂停条件。
5. 接入表现层状态显示，并确保 `Gameplay` / `DebugSandbox` 复用。
6. 补齐 EditMode 和 PlayMode 测试。

回退策略：保留旧的 `TrySelectNearestSafeMineTarget` 兼容测试入口，或将完整自动 tick 通过配置开关关闭；如自动模式异常，机器人仍可保持生产和显示，不影响玩家主循环。

## Open Questions

- 机器人默认钻头能力应固定为土层，还是跟随玩家当前钻头等级？本提案建议先跟随玩家当前钻头等级，保证升级能自然扩大自动化范围。
- 机器人挖掘奖励是否 100% 进入玩家资源？本提案建议 MVP 先即时入账，后续再考虑返航卸货。
- 机器人触发炸药后的爆炸是否应该伤害附近玩家？本提案建议暂不做玩家直接伤害，只做地形爆炸和机器人损毁。
