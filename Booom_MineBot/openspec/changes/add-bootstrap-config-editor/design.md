## Context

当前项目已经将核心可调数值放进 `ScriptableObject`，并由 `BootstrapConfig` 统一聚合注入运行时。这符合仓库“配置优先用 SO、运行时规则与表现层解耦”的方向，但编辑期工作流仍然停留在最原始的 Inspector 拖引用阶段。

这带来三个具体问题：

- `BootstrapConfig` 下面的多个子配置很容易漏挂、错挂，尤其在新增配置类型后更明显。
- 配置入口暴露的是一串对象引用字段，而不是面向策划/设计的“大项配置分类”。
- 建筑定义是数组引用，维护成本比单个 SO 更高，容易出现资产已存在但没同步到数组里的情况。

本次设计继续遵守现有仓库边界：

- 只改编辑期工具，不改运行时规则装配边界。
- 继续沿用 `BootstrapConfig` 作为聚合入口，而不是再引入一层新的数据库或外部表系统。
- 技术栈使用 Unity 6 + `UIToolkit` + 原生 `SerializedObject`/`PropertyField`，不引入第三方编辑器框架。

## Goals / Non-Goals

**Goals**

- 提供一个独立的 `UIToolkit` 配置编辑器窗口。
- 配置列表按启动、地图、成长/经济、挖掘/风险、波次、建筑等大类组织。
- 自动托管 `BootstrapConfig` 下的必需子配置资产，缺失时自动创建并回填。
- 让建筑定义列表支持自动同步和一键新建，避免手动维护数组引用。
- 补充 EditMode 测试，锁定自动创建/引用逻辑。

**Non-Goals**

- 不重做现有运行时配置类型本身的数据结构。
- 不把 authored `MapDefinition` 从“可选入口”改成“必填资产”。
- 不在本轮实现复杂的图形化地图编辑器、节点图或多窗口联动流程。
- 不修改现有场景、Prefab 或运行时服务的装配顺序。

## Decisions

### 1. 使用独立 `EditorWindow`，而不是把所有逻辑塞进 `BootstrapConfig` 自定义 Inspector

本次采用独立 `EditorWindow`，而不是单纯重写 `BootstrapConfig` Inspector，原因有三点：

- 用户明确要求“配置列表按大项分类”，这天然更适合左侧分类、右侧详情的窗口布局。
- 一个窗口可以同时承载“自动补齐引用”“新增建筑配置”“创建地图配置”等跨资产动作，不会把聚合根 Inspector 做得过重。
- 独立窗口更容易扩展到未来的其它配置入口，而不要求用户必须先选中 `Bootstrap.asset`。

备选方案：只做 `BootstrapConfig` 自定义 Inspector。  
放弃原因：分类导航弱、跨资产操作拥挤，而且仍然依赖用户先找到正确的聚合资产。

### 2. `BootstrapConfig` 继续作为聚合根，编辑器只负责自动托管其下游资产

本次不会新增第二层“配置数据库”或外部表。编辑器只负责把现有聚合关系维护好：

- `inputActions`：若为空，自动尝试绑定项目内默认输入资源。
- `balanceConfig` / `upgradePool` / `hazardRules` / `miningRules` / `waveConfig`：若为空，优先复用同目录已有同类型资产；找不到再创建默认资产并回填。
- `buildingDefinitions`：自动收集 `BootstrapConfig` 所在目录树下的 `BuildingDefinition` 资产，并同步回数组。

这样能保证：

- 运行时仍只认现有 `BootstrapConfig`。
- 现有 `Assets/Settings/Gameplay` 资产不需要迁移。
- 自动托管逻辑集中在 Editor 侧，可测试、可复用、可渐进扩展。

### 3. 配置列表按“玩法大类”分组，而不是按资产类型平铺

窗口左侧分类采用以下六组：

- 启动与场景
- 地图
- 成长与经济
- 挖掘与风险
- 波次
- 建筑

右侧每个分类再按需要展示一个或多个配置资产的默认 Inspector 内容。这种分组比“每个 SO 一个条目”更接近项目内的真实配置语义，也符合用户对“大项配置分类”的要求。

### 4. authored 地图保持可选，只提供“一键创建并引用”入口，不自动强制创建

`DefaultMap` 当前为空时，运行时会走程序生成地图。这是已有合法路径，因此不能在窗口打开时无条件创建 `MapDefinition`，否则会改变当前游戏行为。

本次策略：

- 地图分类里继续展示 `defaultMap` 与 `generatedMapConfig`
- 若用户需要 authored 地图，提供“创建/引用地图配置”按钮
- 只有用户显式触发时，才创建并回填 `MapDefinition`

这样既满足“无需手动引用”的目标，又不破坏当前“空地图引用 = 使用程序地图”的既有含义。

### 5. 建筑配置采用“目录扫描 + 一键新建 + 数组同步”

`BuildingDefinition[]` 是当前最容易失配的一组引用。本次不要求用户手动维护数组，而是：

- 扫描当前 `BootstrapConfig` 目录树下的全部 `BuildingDefinition` 资产
- 保留当前数组中已存在的有效引用，避免误丢外部资产
- 若一个都没有，则自动创建一份默认建筑配置
- 提供“新增建筑配置”按钮，在托管目录下生成新资产后立即同步数组

这比继续让用户手动加数组元素再拖对象更稳，也更符合“无需手动引用 SO”的目标。

## Risks / Trade-offs

- [窗口只覆盖 `BootstrapConfig` 管辖的主配置，不是全项目所有配置中心]
  Mitigation：本次明确聚焦启动链路使用的主配置，避免一次性做成无边界工具。

- [自动扫描目录同步建筑引用，可能把同目录下临时建筑资产也纳入]
  Mitigation：当前目录树本来就是聚合配置的合理边界；若后续出现噪音资产，再补更细的命名或目录约束。

- [使用默认 Inspector 组合多个 SO，视觉上不如全手写表单统一]
  Mitigation：本次优先完成稳定的资产托管与分类导航；后续若高频使用，再对单类配置做更细致的定制布局。

## Migration Plan

1. 新增 Editor 侧配置资产托管工具，先把“找/建/回填”逻辑独立出来。
2. 新增 `UIToolkit` 配置编辑器窗口，用大类分类组织现有配置。
3. 为建筑定义增加“一键新建 + 自动同步”。
4. 补 EditMode 测试，覆盖自动创建、地图按需创建、建筑同步。
5. 用 UnityMCP 做编译与测试验证。
